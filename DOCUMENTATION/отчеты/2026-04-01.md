## 1. Pipeline Constructor (Breakdown Studio)

### Визуальный редактор пайплайна
- Полный граф на ReactFlow: модули, данные, связи, ForEach
- Каждый модуль: systemPrompt, модель (Claude/Gemini/GPT), temperature, inputs/outputs
- Drag & drop нод, zoom, pan, minimap
- Путь: `/dev/breakdown-studio`

### Два встроенных пресета

**Fincher (5 модулей):**
1. Scene Analyst → анализ сцены (тон, география, персонажи, пропсы)
2. Prop Scanner → находит ВСЕ предметы, описания + appearance prompt
3. Shot Planner → раскадровка в стиле Финчера (surveillance, геометрия)
4. Prop Distributor → распределяет пропсы по кадрам (что видно, что нет, почему)
5. Prompt Composer → per-shot промпты через ForEach (200-350 символов)

**Анимация / Disney (2 модуля):**
1. Shot Planner → яркая раскадровка (эмоции, динамика, пропсы критичны)
2. Prompt Composer → per-shot через ForEach, пропсы ОБЯЗАТЕЛЬНЫ в промпте

### Сохранение
- Автосейв сессии: текст сцены, Bible, стиль, пресет → localStorage (debounce 1с)
- Картинки Bible + сгенерированные шоты → IndexedDB
- Сохранение/загрузка конфигураций пайплайна по имени

## 3. Prop Distributor (новый модуль)

- Стадия 2.5 между Shot Planner и Prompt Composer (только Fincher)
- Для КАЖДОГО шота решает какие пропсы РЕАЛЬНО видны с данного ракурса
- Учитывает: shot size (ECU → 0-1 проп, Wide → все), camera angle (сверху → стол виден, стена нет), depth (foreground/mid/background)
- Возвращает: visibleProps с placement и size_in_frame + excludedProps с объяснением
- propAdvice — профессиональный совет для Prompt Composer

## 4. Адаптивный Prompt Composer

- `hasRefImage=true` → имя персонажа БЕЗ описания (фото говорит за себя)
- `hasRefImage=false` → краткое описание внешности (15 слов)
- Close-up → пропускает описание локации
- Wide → включает локацию
- Пропсы: large → описание с материалом/цветом, small → только имя
- Бюджет: 40-60 слов (200-350 символов) для Gemini
- Никаких абстрактных слов (tension, mood, atmosphere)

## 5. LLM Reference Image Picker

### Проблема
Gemini получает до 8 референсов, но надо отправлять ТОЛЬКО релевантные для кадра.

### Решение
- Перед генерацией: собирает каталог Bible с индексами
- Отправляет Gemini Flash: "вот промпт, вот каталог — верни индексы"
- Приоритет: пропсы > главный персонаж > второстепенные > локация
- Результат кешируется по промпту
- Pre-pick для всех шотов после Run Pipeline (UI показывает refs до генерации)

### API
- Лимит референсов поднят с 3 до 8 (`/api/nano-banana/route.ts`)
- Nano Banana 2 = Gemini 2.5 Flash Image, спокойно принимает 5-8 refs

## 10. Синхронизация стенд → проект (КЛЮЧЕВОЕ)

### Проблема
Настроили пайплайн на стенде, но в проекте работал старый код. Промпты из пайплайна перезаписывались `buildImagePrompt()`.

### Решение

**ForEach в fincher.ts:**
- `runPipelinePreset()` теперь определяет: если модуль имеет `shotPlan` в inputs и предыдущий результат содержит `shots[]` → запускает per-shot (как на стенде)
- Каждый шот = отдельный вызов LLM

**promptBuilder не перезаписывает:**
- После breakdown: если `activePipelinePreset` с модулями → `buildScenePromptDrafts` пропускается
- Inspector: `shot.imagePrompt || buildImagePrompt()` — промпт из пайплайна приоритетнее
- Генерация: то же правило

**Итоговая цепочка:**
```
Bible → выбрал "Анимация"
  → pipeline активирован
  → стиль = "3D animation..."

STORYBOARD → RE-BREAKDOWN
  → fincher.ts → runPipelinePreset()
    → Shot Planner (1 вызов) → N шотов
    → Prompt Composer (ForEach: N вызовов)
  → промпты записаны
  → promptBuilder НЕ трогает

Generate → промпт из пайплайна + LLM ref picker → Nano Banana
```

## 11. Исправления багов

| Баг | Причина | Исправление |
|-----|---------|-------------|
| Картинки пропадают в Bible | Stale closure в `setBible({...bible})` | Все `setBible` → `setBible(prev => ...)` |
| Генерация не работает | Stale `generatingShots` в useCallback | `useRef` для актуального значения |
| Gemini блокирует промпты | "Disney"/"Pixar" в тексте | Strip перед отправкой + запрет в system prompt |
| ForEach шоты теряются | Batch=3 для Gemini → rate limit | `batch=1` для Gemini моделей |
| Промпты дублируются | `buildImagePrompt` перезаписывает | Skip когда pipeline active |

## Открытые вопросы

1. **Композитный референс** — для сложных сцен с 6+ элементами, коллаж из refs в одну картинку
2. **ForEach на стенде vs fincher.ts** — на стенде батчи параллельные, в fincher.ts последовательные (можно ускорить)
3. **Состояние пропса** — система не различает "открытая шкатулка" vs "закрытая" (нужно несколько вариантов пропа в Bible)
4. **Кастомные пресеты из UI** — пользователь собирает пресет на стенде, но добавить в Bible список может только через код
5. **Старый promptBuilder** — при Fincher пресете (modules=[]) используется дефолтный fincher.ts + promptBuilder. Нужно ли унифицировать?
